## 🚨 Incidente: rotación de la contraseña de producción

La `DATABASE_PUBLIC_URL` de prod quedó expuesta en el chat (al hacer el backup), así que se rotó la
contraseña. **Causó una caída breve de producción** por desconocimiento de un detalle de Railway.

**Línea de tiempo:**
1. `ALTER USER postgres WITH PASSWORD '...'` vía `psql` contra prod → cambió la clave **real**. ✅
2. **Error 1 (usuario):** se cambió la variable `POSTGRES_PASSWORD` en el servicio **equivocado**
   (Postgres-Staging) y se deployó → staging quedó **inconsistente** (variable ≠ clave real).
3. Se sincronizó la variable en el Postgres de **producción** y se redeployó el web.
4. **Caída:** `www.profeonline.cl` devolvió **500** unos minutos — el web seguía con la
   `DATABASE_URL` (referencia) **cacheada con la clave vieja**.
5. **Recuperación:** *Redeploy* del servicio web de prod → tomó la `DATABASE_URL` (referencia
   `${{...}}`) ya resuelta a la clave nueva → **200 OK** restaurado. Datos intactos.
6. **Staging:** se **revirtió** la variable a su valor original (recuperado con `printenv DATABASE_URL`
   desde el contenedor del web, que aún tenía la clave vieja en memoria) → staging consistente y 200.

**Lecciones (ver runbook-backups §5):**
- En Railway, **cambiar la variable `POSTGRES_PASSWORD` NO cambia la clave real**: hay que correr
  `ALTER USER` en la base. Variable y clave real **deben** quedar iguales.
- Tocar la variable **solo en el servicio correcto** (prod ≠ staging).
- Si el web usa **referencia** `${{Postgres.DATABASE_URL}}`, hay que **redeployar el web** para que
  tome la clave nueva (la referencia no se propaga a un proceso ya corriendo).
- Hacer `ALTER` + actualizar variable **casi simultáneo**; asumir una ventana corta de downtime.
